# Product Requirements Document: Tessera

## 1. Executive Summary
Tessera is an opinionated, local-first AI desktop orchestrator designed for business users. It rejects the fragile, open-ended "vibe coding" of standard AI chatbots. Instead, Tessera uses deterministic state-machine architecture to execute repeatable business workflows such as lead syncing, data extraction, document processing, and back-office operations with predictable boundaries. It pairs a secure, lightweight OS shell with mandatory business identity, local data protection, local workflow checkpoints, a strict tool permission model, and a future local memory network to guarantee data privacy and verifiable Human-in-the-Loop execution.

## 2. Target Audience
* **Business Operators & SaaS Founders:** Users who require reliable automation for daily operations without trusting sensitive client data to fully autonomous cloud agents.
* **Privacy-Conscious Professionals:** Consultants and managers who need fully air-gapped or hybrid-local execution capabilities.
* **Workflow Power Users:** Individuals frustrated by having to repeat complex conversational prompts to achieve the same result twice.

## 3. Core Architecture & Stack
Tessera operates on a "Sidecar & CLI" architecture to maintain strict physical and memory boundaries:
* **The Shell:** Tauri v2 (Rust-based, capability-scoped OS boundary).
* **The Orchestrator:** Bun (High-performance sidecar daemon).
* **The State Engine:** XState (Deterministic workflow execution, suspension, and resume semantics).
* **The Checkpoint Store:** Sidecar-owned SQLite database for persisted workflow run state.
* **Identity & Data Protection:** Mandatory Google/Microsoft login first, OS-secure credential storage, and local encrypted/session-locked business data.
* **The Cognitive Loop:** `@mariozechner/pi-agent-core` (Minimalist tool-calling loop).
* **The Tooling:** Pre-compiled standalone CLI binaries and tool adapters acting as the only execution levers.
* **The Git Ledger:** Bundled, version-pinned Git runtime used by Tessera Git Service for workspace history, diffs, rollback, and workflow run branches.
* **Inference Layer:** `@mariozechner/pi-ai` as the provider abstraction layer, supporting OpenRouter and local OpenAI-compatible endpoints such as Ollama or LM Studio.

## 4. Key Capabilities & Product Requirements

### 4.1 Deterministic Workflow Execution
* The system must execute workflows defined strictly as XState JSON/TypeScript state machines.
* The LLM must never dictate the workflow path; it is strictly invoked by XState to process data or select CLI parameters.
* All tool arguments generated by the AI must pass strict Zod schema validation before execution.
* Workflow run state must be checkpointed durably whenever execution blocks, resumes, completes, fails, or is denied.

### 4.2 Identity, SSO, and Local Data Protection
* Tessera must require user login before workflow execution or access to local business data.
* Google and Microsoft SSO are the first supported identity providers.
* Enterprise SSO (SAML/OIDC), centralized audit export/sync, admin policy controls, and org management are later enterprise capabilities.
* Authentication must be part of the local data-protection boundary, not merely a UI gate. Local business data must be encrypted or session-locked to the authenticated user.
* OAuth/session credentials and local encryption material must be stored using OS-secure storage such as the platform keychain.
* Tessera must record local structured audit events for authentication, workflow runs, HITL approval/denial, and tool permission decisions. These events should be designed for later centralized export/sync.

### 4.3 First-Login Onboarding Memory
* After first login, Tessera must run a brief onboarding interview to seed local user memory.
* The onboarding interview may collect role, work context, company/team context at a non-secret level, common workflows, tools used, data sensitivity expectations, approval preferences, writing style, tone, working preferences, and personalization cues.
* Onboarding must rely on explicit volunteered answers. It must not passively scan private files, email, or connected accounts.
* Onboarding must not collect personal life details, health or finance information, company secrets, employee monitoring signals, or performance evaluation data.
* The first onboarding memory seed must be stored locally and later become an input to Hindsight/GBrain once the dual-memory system exists.

### 4.4 The Dual-Memory System
* **Hindsight (Episodic):** An embedded PostgreSQL/`pgvector` daemon running locally without Docker. It stores immutable facts and temporal workflow outcomes for immediate context retrieval (Pre-Flight).
* **GBrain (Semantic):** A background compiler that uses idle system time to synthesize Hindsight facts into a structured, cross-referenced Markdown wiki stored in a local, Git-backed directory.

### 4.5 Action Inbox and Human-in-the-Loop (HITL) Security
* Tessera must implement an Action Inbox instead of a single-purpose approval dialog. Approval is one inbox action type, not the core primitive.
* Tasks, workflow runs, agents, and system services may emit inbox messages when user action is required.
* Inbox messages must be persisted state-machine events, not chat messages or transient notifications.
* Every inbox message must include `id`, `source`, `workflowRunId` or `taskRunId` when applicable, `type`, `severity`, `status`, structured `context`, allowed `actions`, optional `deadline`, and an immutable audit log.
* Supported message types must include `approval`, `input_required`, `review`, `exception`, `credential`, `policy_override`, `prioritization`, `artifact_review`, and `production_promotion`.
* Supported statuses must include `open`, `snoozed`, `resolved`, `expired`, and `cancelled`.
* The orchestrator must classify all CLI tools strictly as "Read" (Safe) or "Write" (Destructive).
* Upon reaching a Write action or any unsafe boundary, XState must suspend execution indefinitely and emit a blocking inbox message containing the exact payload, risk context, permission delta, and allowed user actions.
* Execution cannot proceed from a blocked state without an explicit inbox action that maps to a deterministic workflow transition.
* Workflows must never wait on free-form chat replies as hidden state. User responses must be captured through typed inbox actions and validated payloads.

### 4.6 Workspace Version Ledger
* Every Tessera workspace must be backed by a Git repository. If the selected folder is not a Git repository, Tessera must initialize a private local repository.
* Git must be used as the workspace ledger for file version history, agent change attribution, diff review, rollback, artifact snapshots, and workflow run boundaries.
* Git must not be used for workflow runtime state, inbox state, secrets, credentials, vector memory, or task queues. Those remain in SQLite, OS keychain, memory stores, or artifact storage.
* Tessera must own all Git operations through an internal Git Service. Agents must never execute raw `git commit`, `git reset`, `git push`, `git clean`, or `git checkout`.
* Each workflow run must start from a pre-run snapshot and execute on an isolated run branch such as `tessera/run/<workflowRunId>`. Future multi-agent runs may use isolated agent branches/worktrees.
* Every agent write must be committed by Tessera with structured metadata including workflow run ID, agent/provider, tool adapter, inbox action ID, affected paths, and risk level.
* Users must be able to restore a single file, undo a workflow run, compare versions, accept changes, discard changes, and promote accepted artifacts without seeing raw Git complexity.
* Large binary artifacts such as Office files and images must be handled with content hashes, previews, extracted text/structure sidecars, and diff summaries. Git LFS is deferred until remote collaboration requires it.
* Tessera must bundle a version-pinned, checksum-verified Git runtime and prefer it over system Git. System Git is only a fallback after version validation. If Git is unavailable, Tessera must open workspaces in read-only mode and block agent writes.

### 4.7 Tool Registry and Readiness Gates
* Tessera must provide a small, stable tool registry shape that maps validated workflow steps to CLI-backed or API-backed tool adapters.
* Tool adapters must declare capabilities, required credentials, required binaries, read/write classification, schema contracts, safe mode availability, timeout behavior, and readiness checks.
* Readiness gates must run at the moment of use, not as a heavyweight diagnostics dashboard. Required checks include startup, workspace open, workflow launch, and tool execution.
* App startup readiness must verify app data directory write access, SQLite availability, OS keychain access, and bundled Git availability.
* Workspace readiness must verify folder access, Git ledger initialization, `.tessera/` write access, and secret-exclusion rules.
* Workflow launch readiness must check only capabilities required by that workflow. No workflow may start unless required capabilities are ready or explicitly available in read-only/safe mode.
* Tool execution readiness must validate schema, permission policy, binary/API reachability, timeout configuration, write classification, and rollback availability.
* UI language must use business-facing states such as `Ready`, `Needs setup`, `Needs login`, `Needs repair`, `Unavailable`, and `Read-only mode`.
* First-party tools should stay generic and workflow-oriented. Tessera must not ship a built-in CRM/ERP product surface.
* Domain-specific systems such as CRM, ERP, helpdesk, finance, and project-management automation belong in workflow manifests, plugins, or external integrations rather than the core product.

### 4.8 Standard Skills and Tessera Playbooks
* Tessera must support standard `SKILL.md` ingestion as a first-class capability so users can reuse existing ecosystem skills.
* Tessera must not invent a competing skill standard. `SKILL.md` remains the reusable agent procedure layer.
* Tessera may add `PLAYBOOK.md` as an additional workflow operating layer above skills. `PLAYBOOK.md` must bind skills to a business workflow, define when each skill is used, describe required user decisions, list produced artifacts, define quality gates, and connect steps to Git, Action Inbox, audit, and workflow manifests.
* Workflow packages may include both standard skills and a Tessera playbook. Skills answer how an agent performs a task; playbooks answer how the business workflow is controlled.

### 4.9 The Auto-Compiler ("Record Macro")
* Users must be able to convert a successful open-ended chat session into a repeatable XState workflow.
* A dedicated reasoning model parses the Hindsight session trace, generates candidate XState JSON and related schemas, validates the artifacts through the Workflow Compiler, and registers the workflow only after successful promotion.

### 4.10 Extensible Workflow Tooling
* Tessera must provide a small, stable tool registry shape that can map validated workflow steps to CLI-backed tools.
* First-party tools should stay generic and workflow-oriented. Tessera must not ship a built-in CRM/ERP product surface.
* Domain-specific systems such as CRM, ERP, helpdesk, finance, and project-management automation belong in workflow manifests, plugins, or external integrations rather than the core product.

### 4.11 Workflow Package UI System
* Tessera must support workflow artifacts that contribute menu entries, forms, workflow definitions, permissions, output views, validation schemas, and onboarding/help content.
* Uploaded workflow artifacts must not ship arbitrary executable UI code in the default path. The safe default is declarative UI: JSON Schema, UI schema, output layout schema, labels, helper text, validation rules, and workflow graph metadata.
* The frontend must remain a renderer and interaction surface. It must not contain workflow business logic, permission decisions, tool execution logic, or hidden state transitions.
* Tessera must support two shell modes:
  * **Default Workspace Mode:** user chooses a workspace, sees files, starts tasks, runs workflows, inspects activity, responds to inbox messages, and views outputs.
  * **Workflow App Mode:** installed workflow artifacts contribute app-like menus, forms, workflow launchers, output views, and help screens without bypassing the sidecar or permission model.
* The import screen for workflow packages must show workflow name, publisher, signature status, requested permissions, tools used, generated UI preview, generated workflow graph, test results, and install/promotion actions.
* Import approval must be modeled as an Action Inbox message of type `artifact_review` or `production_promotion`, not a special one-off UI path.

### 4.12 Workflow Compiler and Generated Code Boundary
* PI is not a workflow installer and must never directly install or execute generated workflow code.
* PI may assist as a compiler assistant only after Tessera understands the target interfaces and validation boundaries.
* Generated workflow code is an artifact, not executable truth. It must be validated, tested, sandboxed, and locally signed before production workloads can use it.
* The Workflow Compiler pipeline must be: imported package, validation layer, PI compiler assistant, generated candidate artifact, static checks and sandbox tests, inbox-based user review, signed local compiled workflow, runtime execution through XState.
* Tessera must define stable interface contracts before enabling workflow code generation: inputs, outputs, auth requirements, retry behavior, failure semantics, side effects, tool permissions, checkpoint behavior, and audit events.
* Validation must include schema checks, static analysis, dependency audit, permission diff, deterministic snapshot generation, fixture-based tests, golden-path tests, and failure-path tests.
* Production promotion requires reproducible test results, rollback path, observability hooks, and explicit human or policy-based promotion through the Action Inbox.
* PI may generate JSON Schema forms, XState workflow drafts, Zod validators, output renderer configs, CLI adapter stubs, docs/help copy, and test fixtures.
* PI must not directly execute arbitrary JS/TS, native calls, network calls, filesystem writes, tool registry mutations, permission expansion, or production promotion.

## 5. Implementation Phases
To prevent AI coding assistants from tangling cross-process boundaries, development must be strictly siloed into these consecutive phases.

### Phase 1: The Secure Sandbox (OS Boundary) — Implemented
**Goal:** Establish secure, isolated communication without any AI or LLM logic.
* Initialize the Tauri v2 shell and the React webview.
* Build the Bun sidecar gateway.
* Define the `capabilities.json` explicitly allowing Bun to spawn child processes.
* Create a dummy `workspace-cli` binary. Prove that clicking a button in the UI tells Bun to spawn the CLI, capture stdout, and return the payload securely.
* Use Unix domain sockets on macOS/Linux by default, with TCP only as the current Windows fallback.
* Require per-launch bearer authentication and host/origin checks on the sidecar boundary.

### Phase 2: The Cognitive Engine (Inference & Tooling) — Implemented Backend Slice
**Goal:** Build the isolated AI execution loop in the terminal backend.
* Use `@mariozechner/pi-agent-core` and `@mariozechner/pi-ai` for the backend cognitive loop.
* Support OpenRouter and local OpenAI-compatible endpoints through validated provider configs.
* Implement a hardcoded read/write capability intercept in Bun.
* Validate tool arguments with Zod before execution.
* Return stable `allow`, `ask`, and `deny` permission decisions for later workflow and UI integration.

### Phase 3: Deterministic Orchestration (XState) — In Progress
**Goal:** Remove open-ended planning by forcing the agent into rigid workflows.
* Integrate XState into the Bun sidecar.
* Maintain a demo deterministic workflow that proves read execution, write suspension, and explicit resume.
* Persist workflow checkpoints in SQLite so paused XState workflows survive application reboots.
* Add APIs and UI affordances to list pending checkpointed workflow runs after restart.
* Define the first manual JSON workflow manifest and load it through `WorkflowDefinitionSchema` instead of hardcoding it in core code.
* Add the Tessera Git Service with bundled Git detection, workspace initialization, pre-run snapshots, run branches, step commits, diff generation, and file/run rollback.
* Add readiness gates for startup, workspace open, workflow launch, and tool execution.
* Wire the HITL intercept to the Action Inbox. Blocking workflow states must emit persisted inbox messages that display the exact payload and allowed actions.
* Add APIs and UI affordances for listing, filtering, opening, snoozing, resolving, expiring, and auditing inbox messages.
* Keep checkpointing and inbox transport independent from future memory integration.

### Phase 4: Identity, Local Data Protection, and Onboarding Memory
**Goal:** Make Tessera acceptable for business use by requiring authenticated access, protecting local business data, and seeding user memory.
* Implement mandatory Google and Microsoft login before workflow execution.
* Store OAuth/session credentials and local encryption material in OS-secure storage.
* Lock local app access and local stored business data to the authenticated user/session.
* Add local structured audit logging for authentication, workflow runs, HITL approval/denial, and tool permission decisions.
* Add a first-login onboarding interview that collects work context and personalization preferences.
* Persist the onboarding result as a local memory seed without requiring Hindsight/GBrain yet.
* Keep enterprise SSO, centralized org audit sync/export, admin policy controls, and org management out of this phase while preserving extension points for them.

### Phase 5: The Dual-Memory Integration
**Goal:** Ground the agent in local reality.
* Configure the Tauri installer/boot sequence to manage the `hindsight-local-mcp` embedded Postgres daemon.
* Wire the `Retain` hook (saving XState outcomes to Hindsight) and the `Recall` hook (injecting operational facts into Pi Agent before Step 1).
* Configure the Tauri idle-time detector to trigger the background GBrain wiki compilation script.

### Phase 6: Skills, Playbooks, and Workflow Package UI System
**Goal:** Let installed workflow artifacts safely contribute app-like UX while reusing standard skills.
* Implement standard `SKILL.md` ingestion for workflow packages and local user-provided skills.
* Implement Tessera `PLAYBOOK.md` parsing as the workflow operating guide above skills.
* Implement declarative workflow package import with manifest validation, signature verification, permission display, generated UI preview, workflow graph preview, and test-result display.
* Implement `react-jsonschema-form` in the Tauri UI to auto-generate input forms directly from uploaded workflow manifests.
* Implement declarative output views from output layout schemas.
* Register validated workflow package menu entries and launchers in Default Workspace Mode and Workflow App Mode.
* Route package review, install approval, and production promotion through the Action Inbox.

### Phase 7: Workflow Compiler and Generated Code Validation
**Goal:** Enable PI-assisted workflow generation only behind deterministic validation and sandbox gates.
* Define Workflow Interface Spec contracts for inputs, outputs, auth, retries, failure semantics, side effects, tool permissions, checkpoint behavior, and audit events.
* Build the Workflow Compiler pipeline for candidate artifacts: schema validation, static analysis, dependency audit, permission diff, sandbox execution, deterministic snapshots, fixture tests, golden-path tests, and failure-path tests.
* Locally sign compiled workflow artifacts after validation and user or policy-based promotion.
* Prevent unvalidated generated code from touching production workloads.

### Phase 8: Extensibility & The Auto-Compiler
**Goal:** Make the platform dynamically grow based on proven user actions.
* Build the Auto-Compiler loop: Extract a successful session trace from Hindsight, route it to an OpenRouter reasoning model, generate candidate XState JSON and related schemas, validate through the Workflow Compiler, and hot-reload the Tauri command palette only after successful promotion.

## 6. Explicitly Out of Scope
* Unauthenticated local-only use for business data.
* Built-in CRM/ERP features, schemas, commands, and UI.
* A separate `tessera-cli` CRM product.
* Domain-specific business databases inside Tessera core.
* Third-party business-data OAuth connectors before deterministic workflow, Action Inbox, checkpointing, manifest loading, and identity/data protection are complete.
* Arbitrary workflow-supplied UI code in the default workflow package path.
* PI-generated workflow code running production workloads before interface contracts, validation, sandbox tests, local signing, rollback, and inbox-based promotion exist.
* Workflows waiting on free-form chat replies as hidden execution state.
* Enterprise SSO, centralized audit sync/export, admin policy controls, and org management in the first identity phase.
* Gateway integrations, remote chat control, messaging-platform operation, and scheduled cron workflows in the current scope.
* Hermes-style global profiles in the current scope; Tessera isolation is based on authenticated user plus workspace.
* Slash-command or command-palette parity with Hermes unless later workflow UX evidence requires it.
