# Phase 2 Plan: Cognitive Engine

## Goal

Phase 2 builds the headless cognitive backend for Tessera. It proves that the
Bun sidecar can run an agent turn, call an LLM through a local backend
abstraction, validate tool arguments, execute approved read tools, and route
write tools through a permission policy.

This phase is intentionally backend-only. It does not add the Tauri approval UI,
XState workflow suspension, keychain-backed API key management, or memory
integration.

This plan expands the Phase 2 outline in [tessera_prd.md](./tessera_prd.md).

## Decisions Locked

- Use `@mariozechner/pi-agent-core` for the agent loop.
- Use `@mariozechner/pi-ai` as Tessera's canonical Phase 2 LLM abstraction.
- Treat the PRD reference to `@badlogic/pi-agent-core` as an outdated package
  name.
- Do not integrate the Vercel AI SDK in Phase 2.
- Support OpenRouter through Pi AI's OpenRouter provider.
- Support local models through Pi AI custom OpenAI-compatible model
  configuration, targeting localhost endpoints such as Ollama or LM Studio.
- Keep API keys as development-only environment variables for this phase.
- Do not persist provider credentials or log secret values.

## Architecture

Phase 2 adds a backend-only agent surface to the existing secure sidecar. The
existing bearer token, host allowlist, and transport security remain the outer
boundary.

The main runtime flow:

1. A caller sends a validated agent request to the sidecar.
2. The sidecar creates a model from the requested provider config.
3. The core runtime registers a small hardcoded tool registry.
4. Pi agent core runs one agent turn.
5. Tool calls are validated against Zod schemas before execution.
6. The permission policy evaluates the parsed tool call.
7. The tool either executes, returns an approval request, or is denied.
8. The sidecar returns a normalized result with messages, tool results, and
   permission decisions.

Proposed subsystem boundaries:

- `packages/contracts`: shared Zod schemas and TypeScript types for agent
  requests, responses, tool calls, tool results, permission decisions, and
  approval payloads.
- `packages/core`: model creation, Pi agent wiring, tool registry, permission
  policy, and tool execution interfaces.
- `apps/sidecar`: authenticated HTTP endpoint that invokes the core runtime.

The likely endpoint is:

```http
POST /agent/turn
Authorization: Bearer <launch-token>
Content-Type: application/json
```

The endpoint should be testable without desktop UI involvement.

## Permission Model

Writes must not be blocked unconditionally. Tessera should use a Codex/Claude
style permission policy that can allow trusted operations, ask for approval when
human judgment is needed, or deny unsupported operations.

The policy returns one of:

- `allow`: execute the parsed tool call.
- `ask`: do not execute yet; return an exact approval payload.
- `deny`: do not execute; return a stable denial reason.

Default behavior for Phase 2:

- Read tools are allowed after schema validation.
- Write tools are evaluated by policy.
- Low-risk writes may be allowed only if covered by an explicit in-memory
  session grant.
- Ambiguous writes return `ask`.
- Unsupported or destructive writes return `deny`.

The Phase 2 session grant model is in-memory only. It should be shaped so Phase
3 can replace the transport/UI around it without changing the core policy
semantics.

Supported grant shapes:

- Exact tool-call grant: allow this tool with this parsed payload.
- Tool-level grant: allow this tool for the current sidecar session.
- Deny rule: block known unsafe or unsupported write operations.

An `ask` payload must include:

- tool id.
- parsed arguments.
- capability classification.
- risk metadata.
- command or operation preview.
- stable reason code, such as `write_requires_approval`.

Phase 3 will use this payload as the XState suspension and Tauri approval UI
input.

## Initial Tools

Phase 2 should start with two hardcoded tools:

- `workspace.ping`: a read tool that executes the existing `workspace-cli ping`
  command through the sidecar's CLI execution path.
- `workspace.writeProbe`: a write-classified probe tool that proves permission
  routing without performing a real mutation.

Every tool definition should include:

- stable id.
- human-readable description.
- capability classification.
- Zod argument schema.
- risk metadata.
- executor function.

Tool arguments generated by the agent must be rejected before execution if they
do not match the schema.

## Acceptance Criteria

Phase 2 is complete when:

- The sidecar exposes a protected backend-only agent endpoint.
- The endpoint accepts a prompt and model/provider config reference.
- The runtime can initialize a Pi agent with Pi AI model configuration.
- OpenRouter can be configured through environment-backed credentials.
- Local OpenAI-compatible endpoints can be configured for Ollama/LM Studio-style
  development.
- Read tool calls validate arguments and execute through the existing CLI spawn
  boundary.
- Write tool calls route through the permission policy.
- `ask` and `deny` decisions never call `Bun.spawn`.
- The response shape is stable enough for Phase 3 to consume.
- Frontend package boundaries remain intact: desktop UI imports only
  `packages/contracts`.

## Test Plan

Add focused tests before broad integration work where possible.

Required tests:

- Provider config validation accepts supported OpenRouter and local model
  configs.
- Provider config validation rejects unknown providers and malformed endpoints.
- Tool argument validation accepts valid tool calls and rejects malformed calls.
- Permission policy allows read tools by default.
- Permission policy returns `ask` for ungranted write tools.
- Permission policy allows an explicitly granted write tool.
- Permission policy returns `deny` for unsupported or denylisted write tools.
- Sidecar endpoint returns stable responses for success, validation error,
  provider/config error, permission `ask`, and permission `deny`.
- Write-probe tests prove blocked writes do not reach CLI execution.

Verification commands:

```bash
bun run check
bun run --filter '*' test
```

## Out Of Scope

The following belong to later phases or separate specs:

- Desktop approval UI.
- XState workflow manifests.
- XState suspend/resume.
- SQLite workflow checkpointing.
- Keychain-backed API key management.
- API key settings UI.
- Hindsight/Postgres memory integration.
- GBrain wiki compilation.
- Auto-compiler workflow generation.
- Built-in CRM/ERP implementation.

## Implementation Notes

Keep the implementation small and reversible. Phase 2 should establish durable
contracts and behavior, not a full product surface.

Avoid adding general plugin loading, workflow manifests, or policy persistence
in this phase. The tool registry can be hardcoded while still using the same
shape expected by future manifests.

Do not log prompts, model credentials, bearer tokens, or full external API
payloads. Test fixtures should use synthetic data only.
